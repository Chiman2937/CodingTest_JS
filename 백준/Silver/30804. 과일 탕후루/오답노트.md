# 과일 탕후루 (BOJ 30804) 오답노트

## 문제 요약

N개의 과일이 순서대로 꽂힌 막대기에서, **두 종류 이하의 과일만 남기도록** 연속 구간을 잘라낼 때
남은 연속 구간의 최대 길이를 구한다.
과일 종류는 1~9번이며, 구간의 앞뒤를 자를 수 있다.

---

## 1차 시도 — 투 포인터 (시간 초과 + 무한 루프)

```js
const filePath = process.platform === 'linux' ? '/dev/stdin' : './input.txt';
let input = require('fs')
  .readFileSync(filePath)
  .toString()
  .trim()
  .split(/\r?\n/);

let [n, list] = input;
list = list.split(' ').map(Number);

function solution(n, list) {
  let result = 0;
  let left = 0;
  let right = 0;
  let map = new Map();
  while (right < n) {
    while (map.size > 1) {
      let target = list[left];
      if (map.get(target) === 1) map.delete(target);
      else map.set(target, map.get(target) - 1);
      left++;
    }
    while (map.size < 2 || map.get(list[right]) !== undefined) {
      let target = list[right];
      map.set(target, (map.get(target) ?? 0) + 1);
      right++;
    }
    let sum = 0;
    for (const [key, count] of map) {
      sum += count;
    }
    result = Math.max(result, sum);
  }
  return result;
}

console.log(solution(+n, list));
```

### 한계 / 분석

**문제 1 — 무한 루프**

두 번째 `while` 조건이 `map.size < 2 || map.get(list[right]) !== undefined`이므로,
과일 종류가 2가지 미만인 채로 `right`가 배열 끝에 도달하면 `right < n` 검사 없이 계속 진행된다.
결국 `right`가 n을 넘어 `list[right] === undefined`가 되어 **무한 루프** 또는 잘못된 접근이 발생한다.

**문제 2 — sum 재계산 O(k)**

매 반복마다 `map`을 순회하여 `sum`을 새로 계산한다.
`map`의 크기 k는 최대 9이므로 상수처럼 보이지만, 근본적으로 **포인터 이동과 별개로 불필요한 반복**을 추가한다.
`sum`은 원소 추가/제거 시 +1/-1로 유지할 수 있는 값이다.

---

## 2차 시도 — sum 증분 관리 + 루프 탈출 조건 추가 (통과)

### 핵심 아이디어

- `sum`을 별도로 들고 다니며 원소 추가/제거 시 `+1 / -1`로 갱신 → O(1) 유지
- 두 번째 `while`에 `right < n` 조건 추가 → 무한 루프 방지

```js
const filePath = process.platform === 'linux' ? '/dev/stdin' : './input.txt';
let input = require('fs')
  .readFileSync(filePath)
  .toString()
  .trim()
  .split(/\r?\n/);

let [n, list] = input;
list = list.split(' ').map(Number);

function solution(n, list) {
  let result = 0;
  let left = 0;
  let right = 0;
  let sum = 0;
  let map = new Map();
  while (right < n) {
    while (map.size > 1) {
      let target = list[left];
      sum--;
      if (map.get(target) === 1) map.delete(target);
      else map.set(target, map.get(target) - 1);
      left++;
    }
    while (right < n && (map.size < 2 || map.get(list[right]) !== undefined)) {
      let target = list[right];
      sum++;
      map.set(target, (map.get(target) ?? 0) + 1);
      right++;
    }
    result = Math.max(result, sum);
  }
  return result;
}

console.log(solution(+n, list));
```

### 한계 / 분석

동작은 올바르지만 **`while` 안에 `while` 두 개**가 중첩되어 있고,
두 번째 내부 `while`의 종료 조건(`right < n && (map.size < 2 || map.get(...) !== undefined)`)이
복잡하여 코드 흐름을 한눈에 파악하기 어렵다.

## 3차 시도 — 슬라이딩 윈도우 단순화 (통과 + 가독성 개선)

### 핵심 아이디어

매 반복에서 `right`를 **하나씩** 늘리고, 창 안의 과일 종류가 3개 이상이면
`left`를 줄여가며 조건을 맞추는 구조로 단순화한다.

```
[초기]  left=0, right=0
         ↓
while right < n:
  list[right] 추가, right++, sum++
  while map.size > 2:        ← 종류 초과 시 왼쪽 축소
    list[left] 제거, left++, sum--
  result = max(result, sum)  ← 항상 유효한 윈도우에서 갱신
```

### 변수 설계

| 변수     | 역할                                     |
| -------- | ---------------------------------------- |
| `left`   | 현재 윈도우의 왼쪽 경계 (포함)           |
| `right`  | 다음에 추가할 원소의 인덱스 (exclusive)  |
| `sum`    | 현재 윈도우 크기 (`right - left`와 동일) |
| `map`    | 윈도우 내 과일 종류별 개수               |
| `result` | 지금까지의 최대 윈도우 크기              |

### 최종 코드

```js
const filePath = process.platform === 'linux' ? '/dev/stdin' : './input.txt';
let input = require('fs')
  .readFileSync(filePath)
  .toString()
  .trim()
  .split(/\r?\n/);

let [n, list] = input;
list = list.split(' ').map(Number);

function solution(n, list) {
  let left = 0;
  let right = 0;
  let sum = 0;
  let map = new Map();
  let result = 0;
  while (right < n) {
    map.set(list[right], (map.get(list[right]) ?? 0) + 1);
    right++;
    sum++;
    while (map.size > 2) {
      if (map.get(list[left]) === 1) map.delete(list[left]);
      else map.set(list[left], map.get(list[left]) - 1);
      left++;
      sum--;
    }
    result = Math.max(result, sum);
  }
  return result;
}

console.log(solution(+n, list));
```

## 복잡도 비교

| 방식                      | 시간복잡도 | 비고                                   |
| ------------------------- | ---------- | -------------------------------------- |
| 1차 (sum 재계산)          | O(n · k)   | k = map 크기(최대 9), 매 반복 map 순회 |
| 2차 (sum 증분, 조건 복잡) | O(n)       | left·right 각 최대 n번 이동            |
| 3차 (슬라이딩 윈도우)     | O(n)       | 동일한 O(n), 코드 흐름이 단순          |

N ≤ 200,000이므로 O(n) 풀이가 필수다.
2·3차 모두 시간복잡도는 같지만, 3차는 **"오른쪽 한 칸 확장 → 조건 위반 시 왼쪽 축소"** 패턴을
일관되게 유지해 슬라이딩 윈도우의 의도를 명확히 드러낸다.

## 회고

1. **투 포인터/슬라이딩 윈도우에서 범위 이탈 가드는 필수다.**
   내부 `while`에 반드시 `right < n` 또는 `left < right` 같은 경계 조건을 포함시킬 것.

2. **누적 가능한 값(sum, count)은 증분으로 관리하라.**
   매 반복 재계산은 불필요하고, 원소 추가/제거 시 ±1로 O(1) 유지가 가능하다.

3. **"한 번에 하나씩 확장, 조건 초과 시 축소" 패턴이 슬라이딩 윈도우의 기본형이다.**
   복잡한 중첩 `while` 대신 이 패턴을 먼저 떠올리면 코드가 훨씬 단순해진다.
